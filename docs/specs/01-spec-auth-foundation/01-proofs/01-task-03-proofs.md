# Task 03 Proofs — Auth.js v5 with Drizzle adapter, Google OAuth, and Resend magic-link

## Task Summary

This task proves that Auth.js (NextAuth v5) is fully integrated: the Drizzle adapter is attached to the Neon client, Google OAuth and Resend email magic-link providers are configured, the four Auth.js-managed tables (`users`, `accounts`, `sessions`, `verification_tokens`) have been migrated into Neon, and the spec's required configuration knobs (database session strategy, 30-day session lifetime, custom auth-page paths) are pinned by automated tests. The auth route handler is mounted at `/api/auth/[...nextauth]` and Auth.js receives requests there in dev.

## What This Task Proves

- `next-auth@beta` (v5), `@auth/drizzle-adapter`, and `resend` are installed at the right major versions.
- The Drizzle schema for Auth.js (`db/schema/auth.ts`) was generated by `pnpm db:generate` and applied to Neon by `pnpm db:migrate` — verified by listing tables in the live database.
- `auth.config.ts` (the static, edge-safe configuration) pins:
  - Session strategy = `"database"` (FR: server-invalidatable; not JWT).
  - Session max age = 30 days exactly (FR: 30-day session longevity).
  - Both providers (`google`, `resend`).
  - Custom pages: `signIn = "/signin"`, `verifyRequest = "/check-email"`, `error = "/auth/error"`.
- `auth.ts` composes the static config with the Drizzle adapter and exports `{ handlers, auth, signIn, signOut }`.
- `app/api/auth/[...nextauth]/route.ts` mounts `handlers.GET` and `handlers.POST` at the canonical Auth.js path.
- `lib/auth/index.ts` re-exports the auth helpers for app code.
- `lib/auth/auth.test.ts` pins all four required FRs above as automated tests.
- `pnpm build` succeeds — Auth.js's dynamic API route is correctly marked `ƒ` (server-rendered on demand).
- All other quality gates (`format:check`, `lint`, `typecheck`) still pass; 11 of 11 tests pass.

## User-Handled Follow-ups (Sub-tasks 3.9 + 3.10)

To turn `GET /api/auth/providers` into a passing 200 response and to enable real sign-ins in Tasks 4–6, the user needs to populate three remaining environment variables in `.env.local`:

1. **Generate `AUTH_SECRET`:**

   ```bash
   openssl rand -base64 32
   ```

   Paste the output into `.env.local` as `AUTH_SECRET=...`.

2. **Create a Google OAuth client** at [Google Cloud Console → APIs & Services → Credentials](https://console.cloud.google.com/apis/credentials):
   - Application type: Web application
   - Authorized redirect URIs: `http://localhost:3000/api/auth/callback/google`
   - Copy Client ID → `AUTH_GOOGLE_ID`
   - Copy Client Secret → `AUTH_GOOGLE_SECRET`

3. **Create a Resend API key** at [resend.com/api-keys](https://resend.com/api-keys):
   - Copy the key → `AUTH_RESEND_KEY`
   - Set `EMAIL_FROM` to either Resend's onboarding sender (e.g. `onboarding@resend.dev`) for early dev or a sender on a verified domain for production-ish use.

After these are set, restart `pnpm dev` and re-run the curl test:

```bash
curl -s http://localhost:3000/api/auth/providers | head -1
# expected: a JSON object containing both `google` and `resend` (or `email`) provider entries
```

Tasks 4.0 and 5.0 do not require Google/Resend keys to be set — the UI and middleware can be implemented and unit-tested without them. The keys are only required for true end-to-end sign-in attempts.

## Post-Task-4 Fix: DrizzleAdapter table-name mismatch

During the live sign-in trial after Task 4.0 was complete, a real sign-in attempt produced `?error=Configuration` and the server log showed:

```text
AdapterError → Failed query: select ... from "user" where "user"."email" = $1
NeonDbError 42P01 (undefined_table)
```

**Cause:** `@auth/drizzle-adapter`'s default schema uses singular table names (`user`, `account`, `session`, `verificationToken`). The spec's schema (and our `db/schema/auth.ts`) uses plural names (`users`, `accounts`, `sessions`, `verification_tokens`). Without explicit wiring, the adapter looked for the singular tables and they didn't exist.

**Fix (2 changes, all gates re-green):**

1. `auth.ts` — pass our plural-named tables explicitly to `DrizzleAdapter(db, { usersTable: users, accountsTable: accounts, sessionsTable: sessions, verificationTokensTable: verificationTokens })`.
2. `db/schema/auth.ts` — added `.unique()` to `users.email` to match the adapter's expected schema (its default `user` table has `email` unique). Regenerated migration `0001_charming_echo.sql` and applied to Neon.

**Verification after fix:** a fresh `POST /api/auth/signin/resend` succeeded with HTTP 302 redirecting to `/api/auth/verify-request?provider=resend&type=email` — no AdapterError, no Configuration error, no DB error. The verification token row was written to Neon's `verification_tokens` table; Resend delivered the magic-link email.
